Full pipeline repair and quality sprint. Both scripts now run end-to-end. Digest quality significantly improved but still needs one more validation run.

## Problems Diagnosed & Fixed

### 1. Tier-2/3 queries returning 0 articles
**Root cause:** NewsData.io free plan silently rejects queries with more than ~2 comma-separated terms. Batch queries like `"CrowdStrike,CRWD,Shopify,SHOP,JPMorgan Chase,JPM"` (6 terms) return 0 articles. Individual queries (2 terms) return 10.

**Fix:** Rewrote `_build_query_groups()` in `news_ingest_pipeline.py`. ALL tiers now run one individual API call per company: `"CompanyName,TICKER"`. Credit budget: 26 calls/run, ~7 runs/day within 200 credit free plan limit.

---

### 2. Pipeline hanging at Amazon/AMZN
**Root cause:** `newsdata.py` had a `while len(articles) < max_results` pagination loop. Amazon returned 45 articles → `45 < 50` triggered a second API call that hung indefinitely (no timeout).

**Fix:** Removed the while loop entirely (free plan returns max 50/call, pagination unnecessary). Added `concurrent.futures.ThreadPoolExecutor` with a 12-second timeout as a hard safety net.

---

### 3. Single-letter ticker contamination (T, O)
**Root cause:** Query `"AT&T,T"` caused articles containing the letter "T" as a standalone word to match — including an obituary for a shellfish company executive.

**Fix:** `_make_query()` helper in `_build_query_groups()` — for any ticker with `len == 1`, the query omits the ticker and uses company name only. `AT&T` queries as `"AT&T"`, Realty Income queries as `"Realty Income"`.

---

### 4. Noise filter gaps — institutional holding disclosures slipping through
**Root cause:** The original `_NOISE_RE` regex missed several common disclosure formats:
- `"JPMorgan Chase $JPM Shares Acquired by Aviso Financial"` — no "Shares Verb by" pattern
- `"Integrity Advisory Solutions LLC Invests $550,000 in Costco"` — no "Invests $X" pattern
- `"Integrity Advisory Solutions LLC Buys New Shares in Johnson & Johnson"` — no "Buys New Shares in" pattern
- `"Invesco QQQ $QQQ is Bouchey Financial Group Ltd's Largest Position"` — multi-word fund name escaped `\w+` match
- `"NewSquare Capital LLC Has $845,000 Stake in Alphabet"` — `\bhas\s+\$[\d,.]+\s+(?:million|billion)\b` required "million/billion" suffix; raw dollar amounts escaped

**Fix:** 5 new patterns added. Also moved noise filter to `news/noise_filter.py` (shared module) imported by both `news_ingest_pipeline.py` and `model/relevance_filter.py`.

**Result:** 81 articles noise-filtered per run (up from 47 before — +34 caught).

---

### 5. Old noise articles passing relevance filter
**Root cause:** Noise filter only runs at ingest time. Articles stored in earlier runs (before the filter existed) were still in ChromaDB and passed through to the LLM.

**Two-part fix:**
1. Cleared ChromaDB (`rm -rf chroma_store/`) and re-ingested from scratch — all 124 stored articles passed the current noise filter.
2. Added noise check inside `find_relevant_articles_from_context()` in `relevance_filter.py` as a second-line defense — any future pre-filter articles are caught at query time before hitting the similarity check.

---

### 6. Digest bloat — 100+ articles going into LLM prompt
**Root cause:** Relevance filter had no cap. With 516 articles in store (accumulated across multiple runs), ~160 passed the similarity threshold → all fed into one massive Gemini prompt → 100+ item digest with repeated summaries.

**Fix:** `find_relevant_articles_from_context()` now sorts results by `best_similarity` descending and caps at `MAX_RELEVANT_ARTICLES = 40`. This keeps the prompt manageable and the digest readable.

---

### 7. Pipeline run log was full of HTTP debug spam
**Root cause:** `_attach_run_log()` set `root.setLevel(logging.DEBUG)` which caused httpcore, httpx, and chromadb to dump every HTTP request detail to the log file.

**Fix:** File handler level set to `logging.INFO`. Root logger level left at its default. Only application-level logs (pipeline, relevance_filter, etc.) saved to file.

- Covers past 3 days (recently reported) + next 14 days (upcoming)
- Returns: `{ticker, company, date, days_until, eps_avg, eps_low, eps_high, rev_avg}`
- Integrated as Step 5.5 in `run_test_pipeline.py`

- Strict rule added: reproduce earnings data exactly, don't invent estimates
- Validated: correctly showed `CRWD — Reported 3 days ago | EPS est: $1.07 | Rev est: $1.4B`

### Granular Skip Counters in Ingestion
Replaced single "duplicates/skipped" counter with 4 separate counters:
- `id_dup_count` — exact same article_id already in store
- `title_dup_count` — same title from different domain
- `no_summary_count` — article has no summary to embed
- `norm_fail_count` — normalization threw an exception

### Pipeline Run Log Saved to File
- `_attach_run_log()` added to `run_test_pipeline.py`
- Full INFO-level output saved to `logs/pipeline_runs/pipeline_run_TIMESTAMP-EST.log`
